## 2026-06-22 - [Refactoring N+1 Query in SupplierOrderHandler]
**Learning:** Found an N+1 issue in `SupplierOrderHandler` (inside `api/barcode.go`) where multiple database inserts were performed in a loop (`tx.Exec`) for each generated barcode when ordering copies. Refactored this to use a single bulk insert operation via `tx.CopyFrom` combined with `pgx.CopyFromRows`.
**Action:** Always prefer `pgx.CopyFromRows` for batch database creation or insertion. This drastically reduces database round-trips when processing larger quantities (like bulk ordering of books).
## 2026-07-07 - [Refactoring N+1 Query in ImportStudentsHandler]
**Learning:** Found an N+1 issue in `ImportStudentsHandler` (inside `api/import.go`) where multiple database UPSERTs were performed in a loop (`tx.Exec`) for each parsed row of a student CSV import. `pgx.CopyFromRows` does not support UPSERT (ON CONFLICT DO UPDATE). Refactored this to use a single bulk UPSERT using PostgreSQL's `UNNEST` combined with `tx.Exec` and arrays, whilst deduplicating rows in memory to avoid "ON CONFLICT DO UPDATE command cannot affect row a second time" errors.
**Action:** Always prefer `UNNEST` combined with `tx.Exec` for bulk UPSERTs. Deduplicate the incoming data locally in a map before executing to prevent conflict errors. Ensure Go slice types strictly match target PostgreSQL array types (e.g. `[]int32` for `int[]`).
